2. Typography Standards

The standard mandates Geist via Google Fonts for all body and heading text, overriding system defaults to ensure unity across macOS and Windows.

Heading 1
Module Overview
Heading 2
Learning Outcomes
Heading 3
Instructions

This is standard body text at 1rem (16px). It should be highly legible, using semi-bold weights for emphasis rather than italics. The line-height is set to 1.5 to ensure comfortable reading during long-form widget cases.

For code blocks, data points, arrays, or mathematical formulas, the standard requires a monospace font stack like Profit = Revenue - Expenses to maintain rigid column spacing.

5. Concept Preamble (Splashscreen)

Every interactive widget should open with a concept preamble screen that explains the underlying principle, formula, or framework before students begin interacting. This ensures learners understand what they're practicing and why, not just how to click buttons. The Streaming Wars simulator below is a live example — click "Begin Simulation" to see the transition.

Splashscreen Requirements

MLO tag: Top of screen, shows which Module Learning Outcome(s) the widget addresses. Uses the turquoise label style.

Title: Clear, descriptive widget name. H3 or equivalent, 700 weight.

Concept explanation: 1-3 sentences explaining the underlying principle, framework, or math. This is the "why" — what the student needs to understand before practicing.

Formula or framework box (if applicable): Displayed in monospace font on a neutral-50 background. Shows the math, model, or decision criteria students will apply.

Task description: On a sand background. Tells the student exactly what they'll do in the activity — in plain, action-oriented language.

"Begin Activity" button: Primary button, centered. Transitions to the interactive content. Student can return to the preamble via a "Back to Instructions" secondary button.

5b. Explicit Submit Requirement

Any widget that requires a student to make a selection must use an explicit submit action. Selecting an option must never be the implicit submit — clicking and confirming are two separate steps.

Incorrect — Auto-advance on click

Which factor drives ROA?

Price
Volume ← clicking this immediately advances
Cost

No confirm step — student cannot reconsider

Correct — Select, then submit

Which factor drives ROA?

Price
Volume ← selected, waiting for submit
Cost
Submit

The Rule

Any selection-based widget must have an explicit submit button. This applies to MCQ, decision trees, drag-and-drop, sorting, and any other widget where the student chooses between options. Clicking an option highlights it — a separate primary button confirms and advances.

The submit button should be disabled until a selection is made. This prevents empty submissions and signals to the student that a choice is required before they can proceed.

The button always says "Submit" — for every step, every widget type, every stage of a multi-step flow. Do not use "Continue", "Next", "See Results", or any other label.

Exception — multi-selection widgets: When a widget evaluates multiple simultaneous placements (e.g. all 12 cards in a drag-and-drop sort, or all rows in a ranking exercise), use "Check Answers" instead of "Submit." "Submit" implies a single answer is being locked in; "Check" communicates that a full batch is being evaluated. This is the only permitted alternative label for an evaluation action.

Exception — non-submission actions: Scenario toggles and preset filters (btn-preset) are not submissions — they filter or reframe content in place and do not need a confirm step. Modal close buttons ("Close Results") and navigation actions ("Back to Instructions") are also exempt. The Submit/Check Answers rule applies only to actions that lock in an answer or trigger evaluation.

Button Placement Rules

Primary actions (Submit, Check, Next) are always bottom-right of their content area. The right side is the "forward" position — it's where learners look to progress.

The standard three-button layout is: Previous (left) · Reset (center) · Submit (right). This gives learners navigation, a clean slate option, and the forward action — all in predictable positions. Left = back, center = undo, right = forward.

Not every widget needs all three. Single-step widgets may only need Reset + Submit. Non-interactive explorations may only need Previous + Next. Use the subset that fits, but always keep the same positional logic: backward actions left, forward actions right.

Scenario toggles and presets go above the content they control, as a horizontal button row. They're input selectors, not actions — placing them above signals "choose first, then see the result."

Exception: Splashscreen "Begin Activity" buttons are centered, since the entire screen is a single call-to-action with no competing content.
